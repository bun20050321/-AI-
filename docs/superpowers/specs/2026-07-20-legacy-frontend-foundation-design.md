# 旧前端基础结构设计说明

## 目标

在现有 `前端` Vue 3 + Vite 项目中增加一个简单、清晰、易维护的应用壳层，并提供维修工单、知识文档和 AI 问答三个基础路由页面。默认页面展示后端健康检查结果。保留旧业务页面文件，不删除、不迁移旧逻辑，也不修改 `frontend/rag-kb-web`。

## 当前基础

- 前端技术栈为 Vue 3、Vite、Vue Router 和 Axios。
- 旧应用入口为 `前端/src/main.js`，根组件为 `前端/src/App.vue`。
- 旧路由默认加载 `UserPage.vue`，另有 `MyRepair.vue` 等旧业务页面。
- 后端已有 `GET /api/health`，成功响应为统一结构：`{ code, message, data }`。
- 后端默认运行在 `http://localhost:8081`。

## 应用结构

`App.vue` 作为全局布局组件，包含顶部系统名称、左侧固定菜单和右侧主内容区。主内容区只负责承载 `router-view`，具体页面各自管理内容和状态。

桌面端使用固定宽度的左侧菜单；小屏幕下改为顶部下方的横向菜单，内容区自然向下排列，避免水平溢出。布局使用普通 CSS，不引入新的 UI 框架、图标库或动画依赖。

## 路由与页面

路由表改为以下三个入口：

| 路径 | 页面 | 用途 |
| --- | --- | --- |
| `/repair-orders` | `RepairOrders.vue` | 默认页面，展示维修工单占位内容和后端健康状态 |
| `/knowledge-docs` | `KnowledgeDocs.vue` | 展示知识文档基础占位内容 |
| `/ai-chat` | `AiChat.vue` | 展示 AI 问答基础占位内容 |

根路径 `/` 重定向到 `/repair-orders`。未知路径也重定向到默认页面，保证菜单与主内容始终处于有效状态。现有 `UserPage.vue`、`MyRepair.vue`、`RepairOrder.vue`、`KnowledgeDoc.vue` 和 `AIChat.vue` 文件继续保留，但不进入新的默认路由。

## HTTP 与健康检查

新增 `src/utils/http.js`，集中创建 Axios 实例：

- `baseURL` 只从 `import.meta.env.VITE_API_BASE_URL` 读取。
- 设置合理的请求超时。
- 响应拦截器直接返回响应体，调用页面无需重复访问 `response.data`。
- 请求错误保持 rejected 状态，由页面决定用户可见文案。

默认页面挂载时调用 `GET /api/health`。仅当响应满足以下条件时显示后端已连接：

- `code === 0`；
- `data` 是有效对象；
- `data.status` 有有效字符串值。

成功时显示服务状态和服务名称。请求失败、超时、非零业务码或响应结构无效时，统一显示固定文案 `后端服务未连接`，不向页面暴露技术异常信息。

开发环境变量 `VITE_API_BASE_URL` 更新为后端实际地址 `http://localhost:8081`。环境文件不包含密码或其他数据库连接信息。

## 样式

全局样式集中在 `src/styles.css`，定义颜色、排版、布局和响应式断点。界面保持工作台风格：白色主内容、深色菜单、清晰的激活态和紧凑的信息层级。页面使用普通区块而非嵌套卡片，避免复杂装饰。

三个页面共享一致的标题与内容间距。菜单使用 `RouterLink`，通过路由激活类展示当前页面。所有交互元素保留可见焦点样式，状态文案不只依赖颜色表达。

## 测试与验证

增加 Vitest、Vue Test Utils 和 jsdom 作为开发依赖，并提供以下自动化验证：

- HTTP 实例读取 `VITE_API_BASE_URL`，响应拦截器返回响应体。
- 默认路由正确跳转到维修工单页面，三个菜单路由均可访问。
- 健康接口成功且响应有效时显示连接状态。
- 请求失败或响应无效时显示 `后端服务未连接`。

完成后运行完整前端测试与生产构建。再启动 Vite 开发服务器，分别以桌面端和移动端视口检查菜单、主内容、状态文案与页面切换，确认无重叠和水平溢出。

## 范围外事项

- 不实现登录、注册、权限控制或用户认证。
- 不接入维修工单、知识文档或 AI 问答的完整业务接口。
- 不重写或删除旧业务页面。
- 不修改后端接口、数据库配置或 `frontend/rag-kb-web`。
