## 5. 建议数据库方案

由于当前项目未明确数据库技术栈，建议采用以下保守方案：

| 场景 | 建议 |
| --- | --- |
| 本地开发 | SQLite 或 PostgreSQL |
| 生产环境 | PostgreSQL |
| ORM | 暂不强制；等项目语言和框架确定后再选择 |
| 迁移工具 | 暂不强制；后续可按项目栈选择 Alembic、Prisma、Flyway 或 Knex |
| 向量数据 | 本阶段只记录向量索引元数据，不保存真实向量 |
| 真实向量库 | 留到后续阶段接入 Pgvector、Milvus、FAISS 或 Chroma |

推荐理由：

- PostgreSQL 适合承载结构化说明书数据、元数据过滤、审计字段和后续生产部署。
- SQLite 适合快速本地验证表结构和示例数据。
- 向量字段先以元数据表记录，可以避免本阶段过早绑定具体向量库实现。

## 6. 核心数据范围

后续数据库建模应至少覆盖：

- 产品型号：产品编码、名称、状态、创建时间。
- 产品版本：型号版本、说明书版本、发布时间、生效状态。
- 说明书文档：文件名、文件类型、来源、版本、校验值。
- 说明书章节：章节标题、层级、排序、页码范围。
- 说明书切块：chunk_id、章节、页码、内容、content_hash。
- 来源追溯：chunk 到原始文档、页码、章节和版本的映射。
- 向量索引元数据：索引版本、模型名、维度、构建时间、状态。
- 切块向量元数据：chunk_id、embedding_status、embedding_model、embedding_dimension。
- 维修联系方式：服务范围、联系方式、状态、生效时间。
- 黄金问题：问题、期望引用、所属场景、是否应拒答。
- 数据库连接检查记录：检查时间、数据库类型、结果、错误摘要。

## 7. 示例数据边界

示例数据必须是虚构、非敏感、可公开的数据。

示例数据可以包括：

- 2 个虚构产品型号。
- 每个型号至少 1 个版本。
- 2 份虚构说明书文档。
- 每份说明书至少 3 个章节。
- 每个章节至少 2 个 Chunk。
- 维修联系方式示例。
- 10 个黄金问题示例。

示例数据不得包含：

- 真实客户姓名、电话、地址。
- 真实公司内部资料。
- 真实 API Key、数据库密码或账号密钥。
- 未经批准的产品参数。

## 8. 连接检查边界

连接检查只验证数据库基础状态，不实现业务 CRUD。

连接检查应包含：

- 数据库连接是否成功。
- 数据库版本或方言信息。
- 关键表是否存在。
- 示例数据数量是否符合预期。
- 能否写入一条数据库连接检查记录。

连接检查不得包含：

- 业务新增、业务查询、业务修改、业务删除流程。
- RAG 检索流程。
- LLM 调用。
- 前端接口验证。
- 维修工单业务验证。

## 10. 主要风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 数据库类型迟迟未定 | DDL 和迁移脚本可能返工 | 先用通用关系模型设计，减少数据库方言绑定 |
| 项目框架未确定 | ORM 和连接脚本无法最终定型 | 暂以文档和 SQL 基础结构为主，脚本保留适配说明 |
| 向量库提前绑定 | 后续技术选型受限 | 本阶段只记录向量元数据，不保存真实向量 |
| 示例数据误含敏感信息 | 合规和安全风险 | 仅使用虚构数据，并在文档中标注非敏感要求 |
| 范围扩张到业务 CRUD | 打破数据库专项边界 | 所有任务明确写明不实现业务 CRUD |
| 说明书字段标准不清 | 表结构难以支撑追溯 | 优先固定型号、版本、章节、页码、source_file、content_hash |
